# Smart Clinic Management System - Database Schema Design

This document details the database schema architecture of the Smart Clinic Management System, utilizing a dual-database design strategy (relational SQL with MySQL and document-store NoSQL with MongoDB).

---

## MySQL Database Design

Relational data is structured and stored inside a relational MySQL schema named `cms`. It holds normalized records with constraints and foreign keys to enforce strict transactional boundaries.

### Table: admin
Stores credentials and high-level administrative users responsible for clinical auditing and provider onboarding.
- **id**: `BIGINT`, Primary Key, Auto Increment, Unique, Not Null.
- **username**: `VARCHAR(100)`, Not Null, Unique.
- **password**: `VARCHAR(255)`, Not Null. (Stored securely as hashed text).

### Table: doctor
Stores basic healthcare provider credentials, professional status, and contact information.
- **id**: `BIGINT`, Primary Key, Auto Increment, Unique, Not Null.
- **name**: `VARCHAR(100)`, Not Null. (Validated: Length between 3 and 100 characters).
- **specialty**: `VARCHAR(50)`, Not Null. (e.g., Cardiologist, Neurologist, Orthopedist, Pediatrician, Dermatologist).
- **email**: `VARCHAR(150)`, Not Null, Unique. (Validated: Matches valid regular expression for email).
- **password**: `VARCHAR(255)`, Not Null. (JsonProperty: Write-only, hidden from JSON response scopes).
- **phone**: `VARCHAR(10)`, Not Null. (Validated: Must be exactly 10 digits).

### Table: doctor_available_times
Associates one or more discrete hourly time tags with doctor records representing weekly clinic shifts.
- **doctor_id**: `BIGINT`, Foreign Key → `doctor(id)`, Cascade On Delete.
- **available_times**: `VARCHAR(20)`. (e.g., `'09:00-10:00'`, `'14:00-15:00'`).
- *Constraints*: Composite Unique Constraint on `(doctor_id, available_times)` ensures a provider cannot represent duplicate shift tags in scheduling databases.

### Table: patient
Maintains public registration demographics and portal access details for clinic patients.
- **id**: `BIGINT`, Primary Key, Auto Increment, Unique, Not Null.
- **name**: `VARCHAR(100)`, Not Null. (Validated: Length between 3 and 100 characters).
- **email**: `VARCHAR(150)`, Not Null, Unique.
- **password**: `VARCHAR(255)`, Not Null.
- **phone**: `VARCHAR(10)`, Not Null.
- **address**: `VARCHAR(255)`, Not Null. (Maximum 255 character string length).

### Table: appointment
Represents a scheduled slot matching a patient and their treatment physician for a custom date and hour.
- **id**: `BIGINT`, Primary Key, Auto Increment, Unique, Not Null.
- **doctor_id**: `BIGINT`, Foreign Key → `doctor(id)` (On delete cascades or updates).
- **patient_id**: `BIGINT`, Foreign Key → `patient(id)` (Allows cascading removal or blocks deletion if strict accounting is required).
- **appointment_time**: `DATETIME(6)`, Not Null. (Validated: Must reside in the future relative to transaction date).
- **status**: `INT`, Not Null, Default `0`. (Standard mapped code: `0` = Scheduled / Active, `1` = Completed / Consulted, `2` = Cancelled).

#### Schema Integrity & Deep-Thinking Questions:
- **Cascading Deletes**: If a patient profile is deleted, we cascade-delete their respective appointments to ensure referential integrity. When a doctor profile is deleted, administrative triggers clear their appointments so patients don't show active appointments with non-existent clinical staff.
- **Avoid Overlapping Bookings**: Overlaps are blocked at the Service Tier. By evaluating whether the tuple `(doctor_id, appointment_time)` is already listed as an active scheduled record in the MySQL reservation matrix, overlapping bookings are prevented.

## MongoDB Collection Design

Variable-schema medical entries and health recommendations are managed inside a document-oriented MongoDB collection named `prescriptions` inside the `prescriptions` database.

Storing clinical scripts inside NoSQL allows rapid customization since prescription fields (such as multi-drug variables, customized notes, refills, and associated pharmacy attachments) vary and require flexible evolution over time.

### Collection: prescriptions
Every document within the collection contains details relating to a finalized doctor-patient appointment.

```json
{
  "_id": "6807dd712725f013281e7201",
  "patientName": "John Smith",
  "appointmentId": 51,
  "medication": "Paracetamol",
  "dosage": "500gm",
  "doctorNotes": "Take 1 tablet every 6 hours.",
  "refillCount": 2,
  "pharmacy": {
    "name": "Walgreens SF",
    "location": "Market Street"
  },
  "tags": ["Pain Relief", "Fever Reduction"],
  "_class": "com.project.back_end.models.Prescription"
}
```

#### Document Schema Mapping:
- **_id**: `ObjectId`, Primary Identifying Document Key (Auto-generated by MongoDB engine).
- **patientName**: `String`, Not Null. (Validated: 3-100 characters).
- **appointmentId**: `Long`, Not Null, Foreign Logical Key linking the document back to the master relational record in MySQL.
- **medication**: `String`, Not Null. (Validated: 3-100 characters).
- **dosage**: `String`, Not Null. (Validated: 3-20 characters).
- **doctorNotes**: `String`, Optional. (Validated: Maximum length 200 characters).
- **_class**: Mapping metadata class annotation added by Spring Data MongoDB to parse standard target model deserializations cleanly into Java records.
